```javascript
function onPick(playerIndex) {
  const randomMove = moves[Math.floor(Math.random() * moves.length)];

  const cpuIndex = moves.indexOf(randomMove);
  const loseIndex = (playerIndex + 1) % moves.length;

  document.getElementById("result-text").innerHTML = "You " + result;
}
```

## Adding Statsig

Let‚Äôs now add Statsig so we can start experimenting with our RPS game.

## Implement in Code

Let‚Äôs update our `index.html` so we can dynamically add the buttons at runtime. We‚Äôll change the body to look like this, giving the buttons `div` an id of "actions‚Äù.

```markup
<body>
  <div class="container">
    <p id="computer-move-text">Pick a Move</p>
    <p id="result-text">...</p>

    <div id="actions"></div> <!-- updated line -->

  </div>
</body>
```

Then we‚Äôll update the `scripts.js` file to pull the move set from Statsig and dynamically add the buttons to the DOM. The new setup function should look like the following. We first get a reference to our layer via `getLayer(‚Äúrps_experiments‚Äù)` and then get our moves by calling `layer.get(‚Äúmoves‚Äù, moves)` . The first argument is the name of our moves parameter and the second is a default value should the parameter not be found.

```javascript
let moves = ["‚úä", "üñê", "‚úå"];

(async () => {
  await statsig.initialize(
    "<CLIENT_SDK_KEY>",
    {
      userID: "some_user_id",
    }
  );

  // -- new lines below

  const layer = statsig.getLayer("rps_experiments");
  moves = layer.get("moves", moves);

  const actions = document.getElementById("actions");

  // Dynamically add buttons to DOM
  moves.forEach((val, index) => {
    const button = document.createElement("button");
    button.textContent = val;
    button.onclick = () => onPick(index);
    button.className = "button";
    actions.appendChild(button);
  });

  // --

})();
```

Now if we reload the page, depending on what experiment group your user is in, you will either see no changes, or you will see the new moves displayed.

If you don‚Äôt see any changes, change the userID until you get one that is in the ‚ÄúTest‚Äù group. You can also use a Feature Gate to override what group a given user is in, but that is beyond this tutorial (learn more [here](/experiments-plus/overrides)).

# Extending the Experiment

So we ran an experiment using `getLayer` but so far the flow hasn‚Äôt been that different from the `getExperiment` API. Let‚Äôs add some more parameters to experiment with to show the power of `getLayer`.

## Giving our Opponent a Name

First, we‚Äôll add a new parameter to our layer that will contain names to give to our AI Opponent. On our layers page (Experiments > Layers Tab > rps_experiments), add parameter to the Layer by selecting ‚Äú+ Add Parameter‚Äù. In the parameter creation dialog that appears, enter the values displayed below.

![9](https://user-images.githubusercontent.com/95646168/162344278-f40fc87a-3b34-450c-a692-63e9dcd74e87.png)

Now, in our `scripts.js` file, in the `onPick` function, let‚Äôs add a call to our new `ai_name` parameter. We will reference the same layer as before, but this time our parameter name is ‚Äúai_name‚Äú. We will again provide a default value (‚ÄúComputer‚Äú) to fallback to incase the parameter cannot be found. With our aiName fetched from the layer, we then update the DOM to use our new name.

```javascript
const aiName = layer.get("ai_name", "Computer"); // new line
document.getElementById(
  "computer-move-text"
).innerHTML = `${aiName} picked ${randomMove}`; // updated
```

## Adding a Scoreboard

Similar to our ai_name parameter, we will add a new score_board_enabled parameter to our layer. This time however, we will use a simple boolean, instead of a string.

![10](https://user-images.githubusercontent.com/95646168/162344320-a7b6c545-a63e-41c0-a53a-70e9df8fb4f7.png)

Now, at the top of `scripts.js`, we‚Äôll add a new variable for holding our score. Then, in the `onPick` function, we‚Äôll increment the score on a win and display it in the DOM. We will use our scoreboard_enabled boolean to control whether or not this is displayed to the user.

```javascript
// Top of scripts.js

let moves = ["‚úä", "üñê", "‚úå"];
let score = 0; // new line
```

```javascript
// onPick function in scripts.js

function onPick() {
  // ‚Ä¢‚Ä¢‚Ä¢

  let result = "Won";
  if (cpuIndex === playerIndex) {
    result = "Tied";
  } else if (cpuIndex === loseIndex) {
    result = "Lost";
  } else {
    score++;
  }

  // ‚Ä¢‚Ä¢‚Ä¢

  // new lines added below
  if (layer.get("scoreboard_enabled", false)) {
    document.getElementById("scoreboard").innerHTML = "Your Score: " + score;
  }
}
```

## Running a Second Experiment

Now with these new parameters, we can run another experiment (even if the first is still running) that contains the original moves parameter as well as the newer ai_name and scoreboard_enabled parameters.

Because we are using `getLayer` rather than `getExperiment`, we don‚Äôt need to update our code to explicitly reference the experiment, getLayer automatically returns the correct values depending on which experiment, if any, the user is in.

Similar to our previous experiment, we will go to the Experiments page and add a new experiment (Again, be sure to set our `rps_experiments` layer under Advanced > Add Layer > Layer). Let‚Äôs call this new experiment ‚Äúupdated_features‚Äù.

For this experiment, we will create 3 groups (Control, Test_Pokemon and Test_AI).

Control will be the same as before. Test_Pokemon will contain Pokemon themed emoji and opponent. Test_AI will also have different emoji and opponent name, but will also display the scoreboard. This should result in the setup below.

![11](https://user-images.githubusercontent.com/95646168/162344363-5c574c59-959f-4cfb-bdce-5380488995a4.png)

You can now run the updated_features experiment in parallel with the first alternative_movesets experiment. Depending on what group the user is in, they will be shown a different RPS game than other users. Change the userID if you wish to try our the different experiences.

So our experiment has been running for a few days and it‚Äôs time to check on the results.

Going to the Results tab on our updated_features experiment we can evaluate how our experiment did.
